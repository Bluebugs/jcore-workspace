# J4 Privileged-Architecture Implementation Design (PM0–PM2)

**Status:** Draft / Proposed
**Companion to:** [`design-spec.md`](design-spec.md) (the architecture; this doc is the *how-to-land-it-in-RTL* half)
**Scope:** the concrete implementation of the privileged-architecture substrate in
the canonical `jcore-cpu` core — milestones **PM0–PM2** (the MMU-independent slice).
PM3–PM4 are a deferred follow-on (§7).
**Audience:** RTL implementers landing the work as PRs.

---

## 1. Framing

"J4" is, per [`docs/glossary.md`](../glossary.md), a synonym for the **J32**
baseline: SH-2 + MMU + the SH-4 privileged architecture, SH4-complete, "planned".
The privileged architecture is the "third bucket" ([`design-spec.md` §1](design-spec.md))
that turns the SH-2-class J2 into J32/J4, and it is a hard prerequisite for the MMU.

This document does **not** restate the architecture — [`design-spec.md`](design-spec.md)
owns the SR layout, banking rationale, exception model, and era-review. It records
the **implementation decisions** for landing PM0–PM2 in the real RTL.

### 1.1 Target repo and toolchain (verified 2026-06-15)

- **Target = the canonical `jcore-cpu` submodule** (`mountain-reverie/jcore-cpu`),
  *not* the stale vendored copy under `jcore-soc/components/cpu` (which still
  carries the archived Clojure generator). All file:line citations below are
  `jcore-cpu`-relative.
- **The decoder is generated by the Go tool** `decode/gen-go/` (`cmd/cpugen`).
  Adding/altering an instruction = editing the per-category **TOML** under
  `decode/gen-go/spec/*.toml` and running `make -C decode generate` (Go 1.26+
  only; no JVM). There is a reserved empty `decode/gen-go/spec/sh4/` — the home
  for the new SH-4/priv instruction rows. End-to-end check:
  `decode/gen-go/regression.sh`. **Never hand-edit the generated VHDL.**

### 1.2 Codebase strategy — one config gate

PM0–PM2 are gated behind a **single** generic/config switch, `PRIV_ARCH`
(following the existing `COPRO_DECODE` generic and the `CONFIG_*` `.vhm`
preprocessor precedents). When off, the J2 build is unchanged (regression target
preserved); when on, the whole privileged substrate is present. We deliberately
avoid three independent flags (`MD`/`RB`/`BL`-separable) because the resulting 2³
build combinations are mostly meaningless (e.g. `RB` functional with banking off).
Sub-granularity can be revisited if a concrete need appears.

### 1.3 Why PM0–PM2 are designed as one unit but landed as three PRs

They share three pieces of machinery, so designing them separately is where
integration bugs hide:

1. **The SR datapath** — PM0 widens `sr_t`; `RB` is inert until PM1 drives the
   regfile; `BL` is inert until PM2 gates exception entry. Threading SR once,
   knowing all three consumers, avoids redoing it.
2. **The exception spine** — PM0's privileged-instruction trap flows through the
   *same* entry path PM2 rebuilds. Designed jointly so PM0 doesn't invent an
   ad-hoc trap that PM2 tears out.
3. **One decoder regeneration** — the new `LDC/STC` rows (PM1), and the `RTE`/
   `TRAPA` rewrite (PM2) all land in the same TOML + microcode; designing them
   together lets one regeneration cover the slice.

Landing is incremental: each milestone has its own self-contained test and ships
as its own PR.

## 2. PM0 — SR widening + privileged-instruction trap

### 2.1 The lucky SR-layout alignment

`core/datapath_pkg.vhd:12-19` already defines the SR bit positions as
`T=0, S=1, IMASK=[7:4], Q=8, M=9` — **identical to SH-4's low bits**. Adding
`MD=30, RB=29, BL=28` is therefore purely additive: no existing bit moves, no
`to_slv`/`to_sr` packing collision. The position constants (currently
`range 0 to 9`) widen to `0 to 30`.

### 2.2 Touch-points

| # | Change | File / line |
|---|---|---|
| 1 | `sr_t` record += `md, rb, bl : std_logic` | `core/components_pkg.vhd:24` |
| 2 | `DATAPATH_RESET` SR sets `md=rb=bl='1'` (SH-4 power-on) — **not** `others=>'0'`, which would zero them | `core/components_pkg.vhd:75` |
| 3 | Add `MD/RB/BL` position constants; pack/unpack in `to_slv`/`to_sr` | `core/datapath_pkg.vhd:12`, `core/datapath.vhm:70-81` |
| 4 | Fold privilege violation into the illegal signal: `this.illegal_instr := check_illegal_instruction(this.if_dr) or (privileged(this.if_dr) and not this.sr.md)` | `core/datapath.vhm:312` |
| 5 | Generator emits a `privileged(opcode)` predicate from a new TOML `privileged = true` field on the control-reg `LDC/STC`, `RTE`, … rows | `decode/gen-go/spec/system.toml` + `internal/emit` |

### 2.3 Why it is non-invasive

- Step 4 means PM0 **reuses the entire existing exception path** unchanged:
  `decode/decode_core.vhm:153-154` already turns `illegal_instr=1` into
  `system_op(GENERAL_ILLEGAL)`. No event-sequencer change in PM0.
- The arith/logic SR-update functions (`components_pkg.vhd:268,315`) touch only
  *named* fields, so the new bits don't perturb them.

### 2.4 Self-testable without PM1/PM2

`LDC Rm,SR` is privileged and runs in supervisor, so it can write `MD=0` to drop
to user mode. **Test:** reset (MD=1) → `LDC` a value with MD=0 into SR → attempt
`LDC Rm,VBR` in user mode → observe `GENERAL_ILLEGAL`; the same in supervisor does
not trap. Re-entry to supervisor needs an exception (PM2) or reset — fine for a
trap test.

### 2.5 Deferred to MMU integration

The spec's **segment** gating (P1/P2/P4 require `MD`) is not implementable in the
standalone `jcore-cpu` core — there is no segment decoder until the MMU/SoC. PM0's
standalone deliverable is the privileged-instruction trap; segment-gating lands at
MMU integration. (Recorded so the spec's §6 isolation guarantee is read as
two-phase.)

### 2.6 Open implementation choice

The `privileged()` predicate as a **generated** function (TOML-driven, consistent
with the decoder philosophy) vs. a small hand-written opcode match.
**Recommendation: generated.**

## 3. PM1 — Real register banking + `LDC/STC R*_BANK`

### 3.1 The orthogonality insight

`core/register_file_two_bank.vhd` `bank_a`/`bank_b` are **mirror copies** written
identically (lines 50-51) — the dual-read-port trick, *not* SH banking. SH banking
is a **different axis**, so we leave the mirror mechanism alone and encode the SH
bank in the **register index**.

### 3.2 Touch-points

| # | Change | Where |
|---|---|---|
| 1 | Grow storage by 8 slots for R0–R7 bank-1 copies (`NUM_REGS` generic +8); R8–R15 stay common | `register_file` generics, `components_pkg.vhd:78` |
| 2 | RB-remap on the four reg-address ports: reg# 0–7 → bank-1 region when `SR.RB=1 ∧ MD=1`, else bank-0; 8–15 fixed. Done in the datapath address path — **regfile entity interface unchanged** | datapath reg-address path (decode→regfile) |
| 3 | `LDC/STC R*_BANK` rows addressing the **opposite** bank (explicit access to the inactive bank for context save/restore) | new `decode/gen-go/spec/sh4/bank.toml` |

The mirror-write at lines 50-51 stays: it writes the *remapped* index into both
read-port mirrors — still correct. Forwarding (`read_with_forwarding`) compares
remapped addresses, so it is unaffected. RB becomes *meaningful* here; exception
entry forcing `RB=1` is wired in PM2.

### 3.3 Test (per spec PM1)

Bank-1 R0–R7 writes invisible from bank 0 and vice-versa; `STC Rm_BANK,Rn` reads
the other bank.

## 4. PM2 — SH-4 register-based exception model

The load-bearing change. The current entries are SH-2 stack-based: e.g.
`Interrupt` (`system.toml:10-78`) is a 6-slot sequence with two `@-R15` stack
writes (`ybus=SR`/`ybus=PC`, `ma_op=WRITE`) and a `VBR+U*4` vector read. PM2
replaces that with a register save.

### 4.1 New state

`SPC`, `SSR` (and optional `SGR`) registers in `datapath_reg_t`.

### 4.2 Touch-points

| # | Change | Where |
|---|---|---|
| 1 | Add `SPC`/`SSR` registers; expose for read via the ybus control-reg mux (`SEL_SPC`/`SEL_SSR` alongside `SEL_SR`) | `components_pkg.vhd:43`, `datapath.vhm:165` |
| 2 | New `sr_sel_t` value `SEL_EXCEPTION`: in one step set `MD=1, RB=1, BL=1` (+`int_mask=ilevel` for IRQ) | `decode/decode_pkg.vhd:29`, `datapath.vhm:373` |
| 3 | Rewrite exception entries to drop the two stack-write slots and add `SSR←SR`, `SPC←PC`, `SEL_EXCEPTION` | `system.toml` `Interrupt`/`Error`/`Slot Illegal`/`General Illegal` |
| 4 | Rewrite `RTE`: `SR←SSR; PC←SPC` (no memory access) — replaces the 3-slot stack read | `decode/gen-go/spec/branch.toml` `RTE` |
| 5 | Rewrite `TRAPA #imm` (`system.toml:1155`): set `TRA`, take general exception via the register model | `system.toml` |
| 6 | `LDC/STC SSR`,`SPC` rows | `decode/gen-go/spec/sh4/` |

### 4.3 Effects and the cross-milestone wiring

The exception microcode gets **shorter** (no `@-R15` pushes), and `RTE` collapses
from 3 memory-read slots to a register restore. Step 2's `SEL_EXCEPTION` forces
`RB=1`, which is what makes PM1's banking pay off on the exception fast path —
the reason these three are one design unit.

Vectors stay `VBR+U*4` (the existing SH-2 table) in PM2; the fixed SH-4 offsets
(`VBR+0x100/0x400/0x600`) are PM3.

### 4.4 The one behavioral break

`TRAPA` changes from "push PC/SR to stack" to the register model
([`design-spec.md` §4.3](design-spec.md)). No SH-2 binary relying on stack-frame
exceptions survives — acceptable because J-Core's own software stack is rebuilt
for SH-4.

## 5. Decoder / TOML mechanics

All instruction-side changes route through `decode/gen-go`:

- New rows live in `decode/gen-go/spec/sh4/` (e.g. `bank.toml`, `system_priv.toml`).
- PM0 adds a `privileged = true` instruction field (new vocabulary in the TOML
  schema + `internal/emit` predicate emission).
- PM2 rewrites existing rows in `system.toml`/`branch.toml`; the `+1`-advance
  microcode sequencer (linear runs of ROM addresses, no loops — see
  `[[jcore-cpu-decoder-architecture]]`) comfortably expresses the *shorter*
  exception sequences, so no new sequencer mechanism is needed.
- After every change: `make -C decode generate` then `decode/gen-go/regression.sh`.

## 6. Milestone / PR plan

| PR | Milestone | Gate test |
|---|---|---|
| 1 | PM0 — SR + privilege trap | user-mode `LDC Rm,VBR` traps; supervisor does not |
| 2 | PM1 — banking + `R*_BANK` | bank-1 writes invisible from bank 0; `STC Rm_BANK` reads other bank |
| 3 | PM2 — register exception model | take illegal-insn exception in user mode, inspect `SSR`/`SPC`, `RTE` returns to faulting context in the right bank/mode |

All three behind `PRIV_ARCH`; J2 build (generic off) stays bit-identical and must
pass existing regression on every PR.

## 7. Deferred (follow-on spec)

- **PM3** — fixed SH-4 vector offsets + `EXPEVT`/`INTEVT`/`TRA` MMIO. Carries an
  external dependency: the `0xFF000020` collision with `CPUINFO`/`ASIDR`
  ([`design-spec.md` §4.6](design-spec.md), owned by the P4-map). Low coupling to
  PM0–PM2, mechanical.
- **PM4** — optional `SGR` (SH-4A binary compat), droppable.
- **MMU integration** — segment (`MD`-gated P1/P2/P4) decode; consumes PM0–PM2 as
  substrate.

---

## 8. Open items

1. `PRIV_ARCH` realized as a VHDL generic vs. a `CONFIG_*` `.vhm` define — pick to
   match how the synth/sim configs (`core/cpu_config.vhd`) prefer to thread it.
2. `privileged()` predicate: generated (recommended) vs. hand-written.
3. `SGR` inclusion deferred to PM4; PM2 must not assume it exists.
4. Exact storage index layout for the bank-1 R0–R7 region (contiguous high slots
   vs. interleaved) — an internal regfile decision, settle during PM1.
